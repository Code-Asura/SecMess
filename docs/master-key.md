# Дизайн master key (этап 5)

Документ описывает формат master key, правила офлайн-хранения и базовый операционный процесс ротации.

## 1. Формат ключа

Канонический формат:

```text
smk1.<key_id>.<secret>
```

Правила:

1. Префикс `smk1` означает `SecMess Master Key v1`.
2. `key_id`:
- символы `[a-zA-Z0-9_-]`
- длина `8..64`
- должен быть уникальным для каждой ротации.
3. `secret`:
- URL-safe символы `[A-Za-z0-9_-]`
- длина `32..256`
- генерируется криптографически стойким ГСЧ.

Пример:

```text
smk1.4f6e0c1b0deab123.JuZ8sYF4A0Y-rkS8uYyJZPz6D0o7.....
```

## 2. Обратная совместимость

Старые ключи без формата поддерживаются только для bootstrap-совместимости.
Для всех новых ротаций используй только канонический формат `smk1.<key_id>.<secret>`.

## 3. Правила хранения и передачи (offline-first)

1. Никогда не передавай master key в чатах, тикет-системах или обычной почте.
2. Основную копию храни в офлайн password manager vault с жёстким списком доступа.
3. Держи одну бумажную sealed-копию в физическом сейфе.
4. Не храни master key в коде клиента или заметках на телефоне.
5. В runtime передавай ключ только через env (`KEYGEN_MASTER_KEY`) и ротируй регулярно.
6. Для админских API-запросов отправляй ключ только по TLS через настроенный заголовок (`X-Master-Key` по умолчанию).
7. Не записывай значение master key в логи.

## 4. Политика ротации

Минимальная политика для MVP:

1. Плановая ротация каждые 30 дней.
2. Немедленная аварийная ротация при подозрении на утечку.
3. После ротации старый активный ключ инвалидируется сразу.
4. Новый ключ показывается один раз в API-ответе и сразу сохраняется офлайн.

## 5. Разделение секретов

`KEYGEN_MASTER_KEY` и `KEYGEN_TOKEN_HASH_SECRET` намеренно разделены:

1. Master key защищает админские эндпоинты.
2. Token hash secret защищает хеширование invite-токенов.
3. Ротация master key не должна ломать хеширование токенов и текущий login flow.

## 6. Incident playbook (подозрение на утечку)

1. Выполни ротацию master key: `POST /keygen/admin/master-key/rotate`.
2. Обнови `infra/.env` новым `KEYGEN_MASTER_KEY`.
3. Перезапусти контейнер keygen.
4. Проверь аудит-события: `GET /keygen/admin/audit/events`.
5. Зафиксируй инцидент: timestamp, actor, цепочка `key_id`.
