ТЕХНИЧЕСКОЕ ЗАДАНИЕ
На разработку, адаптацию и развертывание защищенного мессенджера (MVP)
 1. ОБЩИЕ СВЕДЕНИЯ И ЦЕЛИ ПРОЕКТА
1.1. Наименование системы: Закрытый защищенный мессенджер для круга доверенных лиц.
1.2. Цель создания: Организация безопасного коммуникационного пространства с автономным управлением, исключающим зависимость от глобальных платформ и сбор персональных данных.
1.3. Подход к реализации: Создание Минимально Жизнеспособного Продукта (MVP) на базе проверенных Open Source решений для обеспечения максимальной безопасности при ограниченном бюджете и сроках.
1.4. Базовый технологический стек:
*   Серверная часть: Matrix Synapse (развертывание через Docker).
*   Клиентская часть: Форк Open Source клиента FluffyChat (Flutter).
*   Дополнительный сервис: Микросервис авторизации (QR-backend) для генерации и проверки одноразовых токенов.
*   База данных: PostgreSQL.
*   Веб-сервер: Nginx (Reverse Proxy + TLS).

2. ФУНКЦИОНАЛЬНЫЕ ТРЕБОВАНИЯ К КЛИЕНТУ (ANDROID)
2.1. Платформа:
*   Основная целевая платформа: Android (формат поставки: APK-файл для ручной установки без магазинов приложений).
*   Архитектура клиента должна допускать возможность будущей сборки под iOS и Desktop без переписывания ядра.

2.2. Интерфейс и Дизайн:
*   Используется стандартный интерфейс клиента FluffyChat.
*   Обязательная кастомизация: замена иконки приложения и названия на указанные Заказчиком.
*   Отключение лишних функций интерфейса, не входящих в scope MVP (Spaces, публичный поиск комнат, многопользовательские аккаунты, если это мешает безопасности).
*   Разработка собственного дизайна и интеграция спрайтов откладывается на последующие этапы.

2.3. Механизм регистрации и входа (Критическая функция):
*   Полное отсутствие регистрации по номеру телефона, email или паролю.
*   Реализация входа исключительно через сканирование QR-кода, содержащего одноразовый токен приглашения.
*   Логика работы:
    1.  Администратор генерирует одноразовый токен с ограниченным сроком жизни.
    2.  Токен кодируется в QR-код и передается пользователю при личной встрече или по переписке.
    3.  Пользователь сканирует QR камерой приложения.
    4.  Клиент отправляет токен на защищенный бэкенд, который через Admin API Synapse создает учетную запись и возвращает `access_token`.
    5.  Токен приглашения немедленно аннулируется после использования.
*   Автоматическая генерация и сохранение ключей сквозного шифрования (E2EE) на устройстве пользователя в момент первого входа.

2.4. Обмен сообщениями и контентом:
*   Поддерживаемые типы контента: Текст и статические изображения.
*   Функции отправки видео, голосовых сообщений, стикеров и сложного сжатия медиа отключены в рамках MVP.
*   Все комнаты по умолчанию создаются с включенным сквозным шифрованием (E2EE). Создание незашифрованных комнат запрещено.

2.5. Конфигурация подключения:
*   Адрес сервера может быть жестко задан («зашит») в конфигурационном файле приложения при сборке.
*   Наличие функции смены адреса сервера внутри приложения не является обязательным для MVP, при условии предоставления Заказчику полной инструкции по самостоятельной пересборке APK с новым адресом.

3. ТРЕБОВАНИЯ К СЕРВЕРНОЙ ЧАСТИ И ИНФРАСТРУКТУРЕ
3.1. Архитектура развертывания:
*   Использование контейнеризации Docker Compose.
*   Изоляция сервисов: база данных (PostgreSQL) и микросервис авторизации не имеют прямого доступа из внешней сети, взаимодействие только через внутреннюю сеть Docker.
*   Внешний доступ только через Nginx (порты 80/443) с обязательным использованием TLS (Let's Encrypt).

3.2. Настройки безопасности Synapse:
*   `enable_registration: false` (стандартная регистрация отключена).
*   `federation_enabled: false` (федерация с другими серверами Matrix отключена полностью).
*   Принудительное включение E2EE для всех новых комнат.
*   Настройка Rate Limiting для защиты от brute-force атак и спама.

3.3. Система управления доступом (Мастер-ключ):
*   Реализация механизма «Мастер-ключа» (Master Recovery Key) — криптографического токена, предоставляющего полные права супер-администратора (доступ к консоли управления и админ-API).
*   Функция ротации ключей: Система должна позволять действующему администратору сгенерировать новый Мастер-ключ, при этом все ранее выданные ключи автоматически аннулируются и теряют силу.
*   Механизм делегирования прав: передача Мастер-ключа доверенному лицу осуществляется офлайн (физически), вход в систему осуществляется путем ввода этого ключа.

3.4. Хранение данных:
*   История переписки хранится в зашифрованном виде на сервере (стандарт архитектуры Matrix).
*   Ключи дешифровки хранятся исключительно на устройствах пользователей и никогда не передаются на сервер.
*   Синхронизация истории при входе с нового устройства обеспечивается стандартными механизмами Matrix.

4. ТРЕБОВАНИЯ К БЕЗОПАСНОСТИ СЕРВЕРА
4.1. Защита периметра:
*   Настройка Firewall (UFW): открыты только порты 80 (HTTP для редиректа), 443 (HTTPS) и нестандартный порт SSH.
*   Установка и настройка Fail2Ban для блокировки IP при множественных неудачных попытках входа.

4.2. Логирование и мониторинг:
*   Настройка логирования событий аутентификации и системных ошибок.
*   Настройка ротации логов для предотвращения переполнения диска.

5. ТРЕБОВАНИЯ К РЕЗУЛЬТАТАМ РАБОТ И ДОКУМЕНТАЦИИ
5.1. Передаваемые артефакты:
*   Работающий сервер Matrix Synapse, развернутый на VPS Заказчика (или переданный с полными правами).
*   Скомпилированный установочный файл клиента APK (Release build), подписанный ключом Заказчика (или с передачей ключа).
*   Полный исходный код модифицированного клиента (форк FluffyChat) и код микросервиса авторизации, размещенный в репозитории Git с доступом для Заказчика.
*   Конфигурационные файлы (шаблоны `.env`, `docker-compose.yml`, конфиги Nginx/Synapse).

5.2. Документация (обязательно на русском языке):
*   Инструкция по развертыванию: Пошаговое руководство по установке системы на чистый сервер Ubuntu с помощью Docker.
*   Руководство администратора: Инструкция по генерации QR-токенов, управлению пользователями, созданию и ротации Мастер-ключа, отзыву прав доступа.
*   Инструкция по пересборке клиента: Руководство по изменению адреса сервера в коде и компиляции нового APK-файла.
*   План аварийного восстановления: Процедура восстановления системы из резервной копии (Backup/Restore).
